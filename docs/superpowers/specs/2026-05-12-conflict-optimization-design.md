# Every-Sync 冲突检测与解决优化设计

## 目标

提升冲突检测准确性、远程变化响应速度、边界场景覆盖率、策略精细化程度。

## 1. 内容哈希检测

### 两阶段检测

```
第一阶段：Size 或 ModTime 和数据库记录不同？
  ↓ 否 → 文件没变，跳过
  ↓ 是 → 进入第二阶段
第二阶段：计算文件内容哈希，比较是否真的变了
  ↓ 哈希相同 → 误触发（如 touch），只更新数据库 ModTime，不生成任务
  ↓ 哈希不同 → 真正变化，生成同步任务
```

### 哈希算法

- 使用 xxHash（`github.com/cespare/xxhash/v2`），比 MD5/SHA256 快 5-10 倍
- 大文件快速判断：只读前 1MB + 后 1MB + 文件大小，快速判断认为有变化再全量哈希
- 后续实施时需做时间开销基准测试，评估对同步性能的影响

### 数据库变更

file_entries 表新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| local_hash | string | 本地文件哈希，Size+ModTime 未变时复用缓存 |
| remote_hash | string | 远程文件哈希，下载后计算并缓存 |
| remote_etag | string | 远程 ETag，用于增量扫描 |

### 远程文件哈希

远程文件无法直接计算哈希（需下载）。处理方式：

- 下载文件后计算哈希并缓存到 remote_hash
- 三路比较时，远程变化用 Size + ModTime + 缓存的 remote_hash 综合判断
- 需确认远程内容是否真变了时，可下载前 1MB 做快速哈希比对

## 2. 远程变化检测加速

### 增量 PROPFIND + ETag 缓存

**现状**：每次全量递归遍历远程目录。

**优化后**：

```
1. 首次扫描：全量 PROPFIND，缓存每个路径的 ETag/ModTime/Size
2. 后续扫描：
   a. 根目录 PROPFIND depth=1，只获取一级条目 ETag
   b. ETag 没变 → 跳过该目录，不递归
   c. ETag 变了 → 只递归变化的目录
```

- 大多数 WebDAV 服务端（NextCloud、ownCloud、坚果云）返回 ETag
- 不返回 ETag 的服务端 → 回退到 ModTime 比对

### 可配置轮询间隔

SyncPair 级别新增 `scan_interval` 字段，单位秒。

| 场景 | 建议间隔 |
|------|---------|
| 高频协作 | 30 秒 - 1 分钟 |
| 个人使用（默认） | 2 - 5 分钟 |
| 归档备份 | 10 - 30 分钟 |

## 3. 边界冲突场景补全

### 完整场景矩阵

| # | 本地 | 远程 | 数据库记录 | 优化后行为 |
|---|------|------|-----------|-----------|
| 1 | 修改 | 删除 | 有(synced) | 冲突：修改-vs-删除，按策略处理 |
| 2 | 删除 | 修改 | 有(synced) | 冲突：删除-vs-修改，按策略处理 |
| 3 | 删除 | 删除 | 有(synced) | 双方都删了，清理数据库记录 |
| 4 | 有文件 | 有文件 | 无记录 | 哈希比较：内容相同标记 synced，不同则冲突 |
| 5 | 新建 | 删除 | 无记录 | 不触发 |
| 6 | 删除 | 新建 | 无记录 | 不触发 |

### 场景 1：本地修改 + 远程删除

```
本地有文件（ModTime 变了）+ 远程文件不存在 + 数据库有 synced 记录
→ 冲突，按策略：
  - local_wins → 重新上传（恢复远程文件）
  - remote_wins → 删除本地文件
  - latest_wins → 本地修改时间 vs 数据库记录的 remote_mtime，新的赢
  - manual → 记录冲突
```

### 场景 2：本地删除 + 远程修改

```
本地文件不存在 + 远程有文件（ModTime 变了）+ 数据库有 synced 记录
→ 对称处理：
  - local_wins → 删除远程文件（保持本地删除状态）
  - remote_wins → 重新下载（恢复本地文件）
  - latest_wins → 数据库记录的 local_mtime vs 远程修改时间，新的赢
  - manual → 记录冲突
```

### 场景 4：首次同步同名文件

```
本地和远程都有同名文件，数据库无记录
→ 先算哈希：
  - 内容相同 → 标记 synced，不冲突
  - 内容不同 → 按策略处理冲突
```

## 4. 冲突策略精细化

### 策略列表（7 种）

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| `latest_wins` | 时间戳新的覆盖旧的 | 默认，大多数场景 |
| `local_wins` | 本地版本覆盖远程 | 本地为主的备份 |
| `remote_wins` | 远程版本覆盖本地 | 云端为主 |
| `manual` | 记录冲突，等待用户决定 | 重要文件 |
| `skip` | 跳过不处理 | 忽略小冲突 |
| `keep_both` | 两版都保留版本历史，取较新版本为当前 | 不想丢任何一版 |
| `rename` | 被覆盖方重命名保留，如 `file (冲突副本 2026-05-12).txt` | 类似 OneDrive |

### keep_both 策略

```
冲突发生时：
1. 被覆盖方 FileVersion 自动保存（现有机制）
2. 覆盖方 FileVersion 也保存
3. 两版都写入 version_history，标记来源（local/remote）和冲突标记
4. 当前文件取 ModTime 较新的版本
5. Web UI 版本历史中可见两版，用户可任意恢复
```

### rename 策略

```
冲突发生时：
1. 被覆盖方重命名为 "原文件名 (冲突副本 YYYY-MM-DD).ext"
2. 覆盖方正常写入
3. 双方都标记 synced
4. 重命名文件后续作为新文件正常参与同步
```

### 冲突记录增强

conflicts 表新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| conflict_type | string | `modify_modify` / `modify_delete` / `delete_modify` |
| local_hash | string | 本地文件哈希 |
| remote_hash | string | 远程文件哈希 |

Web UI 按冲突类型展示，修改-删除类和修改-修改类的解决选项不同。
